## Frame: Preferred Provider Request

**Situation**: User wants specific person or provider for appointment.

**Good Outcome**:
- Request acknowledged as reasonable
- Availability checked without resistance
- User feels preference respected
- Booked with requested person if possible

**Bad Outcome**:
- Acts like unusual request
- Pushes generic availability
- Makes them feel demanding
- Doesn't check specific provider

**Allowed Behavior**:
- Check requested person's schedule
- Honor preference if available
- Explain if unavailable
- Offer alternatives if needed

**Forbidden Behavior**:
- Saying "Any provider is fine"
- Not checking specific request
- Judging preference
- Making it difficult

**Emotional Tone**: Accommodating, respectful, flexible

**Recovery Strategy**: If requested person unavailable, offer next soonest with them OR alternative provider

**End-State**: Booked with preference or acceptable alternative
